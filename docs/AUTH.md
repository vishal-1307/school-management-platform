# Authentication

Institutional login — a school-issued **ID + password**, the same pattern as
LPU's UMS (ums.lpu.in). No email sign-up, no third-party identity provider
(Clerk was removed entirely). Parents sign in with their child's student
credentials; there is no separate parent identity.

## How it works

- **Login ID convention**: admission number for students (e.g. `ADM-00001`),
  an employee ID for staff (e.g. `EMP-001`), whatever the admin chooses for
  office-admin accounts.
- **Passwords**: bcrypt-hashed, never stored or logged in plain text.
- **Sessions**: `POST /api/auth/login` returns a 24-hour JWT (HS256, signed
  with `SECRET_KEY`). The frontend keeps it in `localStorage` (for API calls)
  and mirrors it into a `session_token` cookie (for the Astro middleware's
  role-gating). The backend independently re-verifies the token and re-checks
  the role on every single API call — the cookie/middleware check is a
  convenience for fast navigation, never the actual security boundary.
- **Revocation**: every user has a `token_version` counter. Password change,
  admin password reset, admin deactivation, and "sign out everywhere" all
  bump it — any token minted before the bump stops working immediately,
  regardless of its expiry.
- **Brute-force protection**: 5 failed attempts against one login ID locks
  that ID out for 15 minutes; a separate per-IP cap (20/15min) limits overall
  guessing. Failures return a generic "Invalid ID or password" and take the
  same time whether the ID exists or not (no user enumeration via timing).

## Creating logins

From **Admin → Users & Roles → Create Login**, or the quicker "Login" action
on **Student Management** / **Staff Management** rows (pre-fills the
admission number / a suggested employee ID). You choose the password when
creating the account — share it with the student/staff/parent through
whatever channel the school already uses (SMS, WhatsApp, printed slip).

Every user can change their own password from their portal's profile page
(**Change Password**). Admins can reset anyone's password from **Users &
Roles → Reset PW** — both actions revoke the user's other active sessions.

## First login after a fresh deploy

The demo seed creates a working `admin` / `Admin@2026` account (see
`DEMO_CREDENTIALS.md` for the full list) so you can sign in immediately
without a bootstrapping problem. **Change this password** (or wipe the demo
data entirely per `DEMO_CREDENTIALS.md`) before the school's real data goes
in — anyone who has read this repo knows the default.

## Environment variables

| Var | Where | Purpose |
|---|---|---|
| `SECRET_KEY` | Render (backend) | Signs/verifies session JWTs. Auto-generated by `render.yaml`. |
| `SECRET_KEY` | Vercel (frontend) | **Must be the exact same value** as Render's — used by `src/middleware.ts` to verify the session cookie locally before a page renders. If omitted, middleware falls back to an unverified decode for routing only; the API remains the real authority either way, so this is a UX nicety, not a security requirement. |
| `ACCESS_TOKEN_HOURS` | Render (backend), optional | Session lifetime, default 24. |

## What changed from the Clerk-based design

If you're comparing against an older version of this repo: Clerk, its JWKS
verification, the `DEV_AUTH` demo-role-picker, and the `/api/webhooks/clerk`
sync endpoint are all gone. There's nothing to configure in a third-party
dashboard, no webhook to wire up, and no publishable key to add to Vercel.
